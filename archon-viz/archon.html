A workflow engine for AI coding agents

Archon turns the squishy "ask the AI to fix this bug" loop into a deterministic, repeatable pipeline. You describe development processes — plan, implement, validate, review, PR — as YAML workflows. A DAG executor runs them, each in an isolated git worktree, calling out to AI providers (Claude Agent SDK, Codex) at the steps where AI actually adds value.

Think Dockerfiles for infra or GitHub Actions for CI — but for AI coding. Same workflow, same sequence, every time. Composable deterministic nodes (bash, git, tests) sit alongside AI nodes (planning, code-gen, review) and human approval gates.

Big-picture architecture

What you get out of the box

📐 Deterministic structure

Workflow YAML defines phases, gates and artifacts. The AI fills in the intelligence at each step, but the structure is owned by you and checked into git.

🌿 Isolated runs

Every workflow run gets its own git worktree (or container — pluggable provider). Run several fixes in parallel without conflicts.

🔁 Loops & approval gates

Loop nodes iterate until a condition is met (ALL_TASKS_COMPLETE, tests pass…). Approval nodes pause execution and wait for a human decision.

🔌 Multi-platform front-ends

Same workflows run from CLI, Web UI (Vite + React), Slack, Telegram or GitHub. Adapters per chat/forge platform feed the same orchestrator.

Workflow node schema (DAG)

A flat Zod schema with superRefine validates all fields, then a transform produces one of six concrete variants. YAML nodes carry no type discriminant — the variant is inferred from which fields are present.

Six DAG node variants

  • CommandNode runs a slash-command-style prompt
  • PromptNode free-form prompt to the AI provider
  • BashNode deterministic shell — no AI involved
  • LoopNode iterates until a condition holds
  • ApprovalNode pauses for a human gate
  • CancelNode aborts the run with a reason

Trigger rules

Each node specifies how its depends_on upstreams must complete:

  • all_success — default; every parent must succeed
  • one_success — at least one parent succeeded
  • none_failed_min_one_success — partial fan-in
  • all_done — finalisers; run regardless of outcome

Common node fields

  • id stringrequired — unique within workflow
  • depends_on string[] — node IDs this one waits for
  • trigger_rule enum — see above
  • prompt string — natural-language instruction (PromptNode)
  • command string — slash command (CommandNode)
  • bash string — shell command (BashNode)
  • loop object
    • prompt string
    • until string — sentinel produced by the AI
    • fresh_context bool
    • interactive bool — pauses for human input
    • max_iterations number
  • retry object — exponential backoff config
  • hooks object — pre/post-step hooks
  • thinking enum|object — Claude SDK thinking config
  • model string — override workflow-level model
  • provider enum — claude | codex
  • sandbox object — permission/tool restrictions
  • condition expr — skip-if predicate evaluated by condition-evaluator

Database — 22 SQL migrations

SQLite (Bun's built-in driver) with hand-written migrations under migrations/. Each migration is numbered and applied at boot via packages/core/src/db/connection.ts. The schema evolved across the project's history — you can read the changelog from the migration sequence below.

    Tables (DAO modules)

    codebases

    Registered project repos. Tracks default cwd, env-var allow-list, isolation policy.

    conversations

    Chat threads — one per platform conversation (CLI session, web tab, Slack thread).

    messages

    Persisted chat turns plus tool-use payloads. Streamed live via SSE.

    sessions

    Immutable AI provider sessions. Migration 010 made them immutable to prevent accidental mutation.

    workflows

    Materialised workflow definitions, including last-activity timestamp and resume path.

    workflow_runs

    A single execution of a workflow. Carries status, ended-reason, dispatched-in-background flag.

    workflow_events

    Append-only event stream per run — drives the UI timeline and post-mortem replay.

    isolation_environments

    Worktrees / containers / future isolation backends. Provider-pluggable.

    codebase_env_vars

    Per-codebase environment variables exposed to workflows. Allow-listed.
